# customer-reply-drafter

**用途**：基于客户上下文生成可审校的英文回复草稿。

## 输入假设

- `files[]`：客户邮件原文、聊天记录、报价单等上下文。
- `taskType = 'reply'`。
- `goal`：必须明确目标（「发送确认」、「提出变更」、「索赔」等），否则草稿方向会偏。
- `riskLevel: high`：本 skill 承诺「未经人工审核不得直接对外发送」，理由是草稿中极易出现商务承诺。

## 输出契约

- `英文回复草稿`：完整段落，带占位符 `{{...}}` 标注用户需要替换的变量。
- `中文备注`：对每段草稿的说明与兜底语境。
- `缺失信息清单`：草拟时发现的知识缺口（价格/交期/认证/付款/物流等）。
- `待确认承诺项`：主管必须检查并签字确认的业务承诺。

## 已知限制

- **不会**直接发送邮件；仅提供草稿 + 待确认承诺项。
- 不保证汇率、交期、MOQ 等数字准确；这些都会出现在 `待确认承诺项` 里要求人工核对。
- 对历史长对话，会做摘要；摘要可能丢失细节，建议主管比对关键数字。

## 失败模式

- 上下文完全缺失（只给一句「写个回复」）：返回空草稿 + `缺失信息清单` 明确说需要补什么。
- 多份冲突材料（如客户两封邮件对同一问题说法不一）：草稿会标注冲突并要求选择主线。

## 升级路径

1. 与 `comment-merger` 串联：主题分组 → 草稿段落分布。
2. 引入模板库（按客户/产品类型），让「待确认承诺项」更结构化。
3. 对接主管审核流：草稿必须走一轮 review 才能落到外发系统。
